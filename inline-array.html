Swift 6.2

InlineArray<N, T> · Array en el stack

Un array de tamaño fijo que vive en el stack, sin heap, sin ARC. Cuando sabes exactamente cuántos elementos tienes.

Antes — no había arrays de tamaño fijo cómodos

Array en Swift siempre vive en heap. Para tamaños fijos la única opción cómoda era una tupla, pero las tuplas son incómodas de iterar, de indexar, e imposibles de usar en genéricos. reserveCapacity tampoco ayuda: es una sugerencia al runtime, no una garantía en compilación.

Swift 5.x — el problema

// Swift 5.x — necesitabas tuplas para tamaño fijo en stack
typealias Matrix4x4 = (
    Float, Float, Float, Float,
    Float, Float, Float, Float,
    Float, Float, Float, Float,
    Float, Float, Float, Float
)
// Incómodo de iterar, de indexar, imposible en genéricos

Ahora — InlineArray<N, T>

El N es parte del tipo. El compilador lo conoce en compilación, puede verificar límites, y el array completo vive en el stack. Cero allocations en heap, cero ARC.

Swift 6.2 — la solución

// Swift 6.2 — InlineArray: tamaño fijo, vive en el stack
var matrix: InlineArray<16, Float> = .init(repeating: 0)
matrix[0] = 1.0
matrix[5] = 1.0   // diagonal de identidad

// El tipo lleva el tamaño — el compilador verifica límites
func initIdentity() -> InlineArray<16, Float> {
    var m = InlineArray<16, Float>(repeating: 0)
    m[0] = 1; m[5] = 1; m[10] = 1; m[15] = 1
    return m
}

// Útil en kernels de imagen, audio, ML inference
struct AudioFrame: BitwiseCopyable {
    var samples: InlineArray<512, Float>
}
// Cero allocations, cero ARC, stack puro

La regla: Ideal para Metal, CoreAudio y ML donde procesas frames de tamaño conocido. Combina bien con BitwiseCopyable. La diferencia con Array: InlineArray no puede crecer, pero a cambio tiene cero overhead de heap.

InlineArray vs Array

No es un reemplazo de Array. Es la herramienta correcta cuando el tamaño es fijo y conocido en compilación, y cuando el rendimiento importa.
