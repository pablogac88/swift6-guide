Swift 6.0

Mutex · Synchronization

El dato vive dentro del lock. No hay forma de tocarlo sin adquirirlo primero.

Antes — el lock y el dato eran dos objetos separados

NSLock, os_unfair_lock, DispatchQueue... todos tenían el mismo problema: el estado protegido y el mecanismo de protección eran dos cosas distintas. Nada impedía acceder al dato sin pasar por el lock.

Swift 5.x — el problema

// Opción 1 — NSLock (Objective-C legacy)
class Cache {
    private var storage: [String: String] = [:]
    private let lock = NSLock()

    func save(_ value: String, key: String) {
        lock.lock()
        defer { lock.unlock() }  // fácil de olvidar
        storage[key] = value
    }

    func otraFuncion() {
        storage["x"] = "y"  // ← acceso directo sin lock → race
        // El compilador no dice nada
    }
}

// Opción 2 — DispatchQueue serial
class Cache2 {
    private var storage: [String: String] = [:]
    private let queue = DispatchQueue(label: "cache")

    func save(_ value: String, key: String) {
        queue.sync { storage[key] = value }  // correcto
    }

    func otra() {
        storage["x"] = "y"  // ← race silencioso, sin warning
    }
}

Ahora — el dato vive dentro del lock

Mutex<Value> encapsula el estado dentro del lock. No existe ninguna forma de leer o escribir el dato sin pasar por withLock. No es un acuerdo de honor — es la estructura del tipo.

Swift 6.x — la solución

import Synchronization

final class Cache: @unchecked Sendable {
    // El dict vive DENTRO del Mutex.
    // No hay ninguna forma de acceder a él sin withLock.
    private let storage = Mutex<[String: String]>([:])

    func save(_ value: String, key: String) {
        storage.withLock { state in
            state[key] = value
        }
        // withLock: adquiere el lock → ejecuta → libera
        // aunque haya un error, el lock se libera igual
    }

    func value(for key: String) -> String? {
        storage.withLock { $0[key] }
    }
}

La regla: La diferencia fundamental: con Mutex es físicamente imposible tocar el dato sin adquirir el lock. El compilador garantiza el acceso, no tu memoria.

¿Cuándo usar Mutex en lugar de actor?

Usa actor cuando puedas. Usa Mutex cuando necesites sincronización síncrona — dentro de un init, deinit, o código que no puede tener puntos de suspensión async. También es la opción natural para interoperabilidad con código C o C++.
