Swift 6.3

@specialize · @inline

Hints al compilador para código performance-crítico. Para autores de librerías y hot paths.

Antes — las funciones genéricas tenían overhead de indirección

Las funciones genéricas en Swift se compilan una vez usando witness tables — tablas de punteros a las implementaciones de los protocolos. En cada operación hay una indirección. El CPU no puede optimizar tan agresivamente como con tipos concretos.

Sin @specialize — overhead de witness table

// Sin @specialize — una sola versión genérica con witness table overhead
func sum<T: Numeric>(_ values: [T]) -> T {
    values.reduce(.zero, +)
    // En runtime: + se llama a través de una witness table
    // Indirección → el CPU no puede hacer SIMD, loop unrolling, etc.
}

// Para todos los tipos → misma versión, mismo overhead
sum([1, 2, 3])          // Int → witness table
sum([1.1, 2.2, 3.3])    // Double → witness table

Ahora — @_specialize e @inline

@_specialize genera una versión del código especializada para un tipo concreto — sin witness table, sin indirección. @inline(__always) incrusta el código en el sitio de llamada. @inline(never) hace lo contrario para error paths.

Swift 6.3 — la solución

// @_specialize: genera versión especializada para Int
// La versión genérica sigue existiendo para otros tipos
@_specialize(where T == Int)
func sum<T: Numeric>(_ values: [T]) -> T {
    values.reduce(.zero, +)
}

// Para Int → código especializado, sin indirección, el CPU puede optimizar
// Para Double → versión genérica normal
sum([1, 2, 3])       // → versión especializada Int
sum([1.1, 2.2])      // → versión genérica Double

// @inline(__always): el código se inserta en el sitio de llamada
// Sin overhead de call/ret. La función no existe como símbolo en el binario.
@inline(__always)
func square(_ x: Int) -> Int { x * x }

// @inline(never): nunca se incrusta, siempre es una llamada real
// Útil para error paths — no contamina los hot paths con código de error
@inline(never)
func handleCriticalError(_ error: Error) {
    // Este código nunca se mezcla con el hot path del caller
}

La regla: Estas herramientas son para autores de librerías y código performance-crítico. En app code normal el compilador ya hace un buen trabajo con whole-module optimization.
