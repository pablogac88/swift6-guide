Swift 6.2

Span<T> · Vistas seguras de buffer

Acceso a buffers de memoria sin copiar ni tomar ownership. La alternativa segura a UnsafeBufferPointer.

Antes — UnsafeBufferPointer: funcional pero peligroso

Para leer un buffer sin copiarlo necesitabas UnsafeBufferPointer — unsafe por nombre y por naturaleza. Nada impedía que el puntero apuntara a memoria liberada, y nada impedía acceder fuera de rango en builds de Release.

Swift 5.x — el problema

// Swift 5.x — UnsafeBufferPointer: funciona pero es unsafe
func processPixels(_ ptr: UnsafeBufferPointer<UInt8>) {
    // Nada impide que ptr.baseAddress apunte a memoria liberada
    // Nada impide acceder fuera de rango en Release
    for i in 0..<ptr.count {
        _ = ptr[i]
    }
}

Ahora — Span<T>: vista sin ownership, con límites

Span<T> es una vista no-owning con límites verificados. El compilador garantiza, mediante dependencias de lifetime, que el buffer origen vive mientras el Span existe. Sin copias, sin ownership, con seguridad.

Swift 6.2 — la solución

// Swift 6.2 — Span: sin ownership, con límites, sin copias
func processPixels(_ span: Span<UInt8>) {
    // Acceso indexado verificado
    // El compilador garantiza que el buffer origen vive aquí
    for i in 0..<span.count {
        _ = span[i]
    }
}

// Crear un Span desde un Array (cero copias)
var pixels: [UInt8] = loadImageData()
pixels.withUnsafeBufferPointer { buffer in
    let span = Span(buffer)
    processPixels(span)
}

// RawSpan: para buffers sin tipo concreto (bytes crudos)
func parseHeader(_ raw: RawSpan) -> UInt32 {
    raw.loadUnaligned(fromByteOffset: 0, as: UInt32.self)
}

La regla: Span es non-escaping por diseño — no puedes guardarlo más allá de su contexto. Esto es exactamente lo que garantiza que el buffer origen siempre vive mientras lo usas. Es el puente entre Swift seguro y código de alto rendimiento.

Span vs Array vs UnsafeBufferPointer

Cada uno tiene su lugar. Span cubre el hueco entre la seguridad de Array y el rendimiento de UnsafeBufferPointer.
