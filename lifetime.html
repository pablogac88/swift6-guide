Swift 6.3

@lifetime · Dependencias de tiempo de vida

Declara que un valor no puede vivir más que otro. La pieza que une ~Copyable y Span en código genérico.

Antes — el compilador no podía verificar la dependencia

En Swift 6.0 y 6.1 podías usar Span y ~Copyable, pero no había forma de expresar formalmente "este Span depende de este Array". El compilador infería algunas dependencias localmente, pero en genéricos y en firmas de funciones públicas se perdía la información.

Sin @lifetime — dependencia no verificada

// Sin @lifetime — el compilador no puede verificar la dependencia
func firstSpan(_ array: [Int]) -> Span<Int> {
    // ¿Cuánto vive este Span? No está claro en la firma.
    array.withUnsafeBufferPointer { Span($0) }
    // Potencial dangling reference: array podría deallocarse
}

Ahora — @lifetime hace la dependencia explícita

@lifetime(param) anota que el resultado hereda el lifetime del parámetro. El compilador verifica que el valor resultante no escapa más allá de lo que permite el parámetro del que depende.

Swift 6.3 — la solución

// Swift 6.3 — @lifetime hace la dependencia explícita
@lifetime(array)
func firstSpan(_ array: borrowing [Int]) -> Span<Int> {
    // El resultado vive como máximo lo que vive `array`
    // El compilador rechaza cualquier escape más allá
    array.withUnsafeBufferPointer { Span($0) }
}

// Con ~Copyable
struct FileHandle: ~Copyable {
    private var fd: Int32

    // La vista vive mientras vive el FileHandle
    @lifetime(self)
    borrowing func bytes() -> RawSpan { ... }
}

let handle = FileHandle(path: "/tmp/data")
let view = handle.bytes()
// `view` no puede escapar más allá del lifetime de `handle`

La regla: @lifetime es la columna vertebral del código de alto rendimiento en Swift 6.3. Une ~Copyable (sin copia), Span (vista sin ownership) y borrowing (sin transferencia). Juntos permiten escribir código tan eficiente como C, pero con verificaciones en compilación.

Las tres piezas juntas

~Copyable, Span y @lifetime son conceptos independientes que se potencian entre sí. Cada uno resuelve una parte del problema de código de alto rendimiento con seguridad.

ConceptoGarantiza
~Copyableel valor no se duplica — existe exactamente una vez
Spanvista sin ownership — no copia, no retiene
borrowingacceso sin transferir — el caller sigue siendo el dueño
@lifetimela vista no escapa más allá del dueño — sin dangling references
