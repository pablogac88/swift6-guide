Swift 6.0

Typed throws

Las funciones declaran exactamente qué tipo de error lanzan. El catch recibe el tipo directamente, sin casting manual.

Antes — throws borraba el tipo del error

En Swift 5, throws siempre era throws(any Error). El compilador no sabía qué podía fallar. El catch obligaba a castear manualmente y a añadir un caso genérico que nunca se ejecutaba.

Swift 5.x — el problema

// Swift 5 — throws lanza any Error, tipo borrado
func parseRecord(from string: String) throws -> String {
    guard string.hasPrefix("REC:") else {
        throw ParseError.invalidHeader  // tipo concreto...
    }
    return String(string.dropFirst(4))
}

do {
    let record = try parseRecord(from: input)
    print(record)
} catch let error as ParseError {  // ← casting manual siempre
    switch error {
    case .invalidHeader:   print("cabecera inválida")
    case .invalidPayload:  print("payload inválido")
    }
} catch {
    // Este bloque era OBLIGATORIO aunque nunca se ejecutara.
    // El compilador no sabía que solo podía llegar ParseError.
    print("Error inesperado: \(error)")  // dead code real
}

Ahora — throws(ConcreteType)

Con throws(ParseError), el compilador sabe exactamente qué puede fallar. El catch recibe ParseError directamente. No hay casting, no hay bloque genérico obligatorio.

Swift 6.x — la solución

enum ParseError: Error {
    case invalidHeader
    case invalidPayload(String)
}

// throws(ParseError) — el compilador sabe exactamente qué puede salir
func parseRecord(from string: String) throws(ParseError) -> String {
    guard string.hasPrefix("REC:") else {
        throw .invalidHeader  // tipo inferido del contexto, sin ParseError.
    }
    let payload = String(string.dropFirst(4))
    guard !payload.isEmpty else {
        throw .invalidPayload("payload vacío")
    }
    return payload
}

do {
    let record = try parseRecord(from: "REC:Hola")
    print(record)
} catch {
    // error ES ParseError directamente — sin casting
    // No hay catch genérico obligatorio
    switch error {
    case .invalidHeader:          print("cabecera inválida")
    case .invalidPayload(let msg): print("payload: \(msg)")
    }
}

La regla: throws sin tipo equivale a throws(any Error) — comportamiento Swift 5. Una función no-throwing equivale a throws(Never). Con tipos concretos, el compilador puede optimizar más y tú escribes menos.

En genéricos — propagación del tipo de error

La novedad más potente es en funciones genéricas. Antes era imposible propagar el tipo de error de un closure — se borraba a any Error. Ahora E es un parámetro genérico del error.

Swift 6.x — genéricos con typed throws

// Antes (rethrows): el tipo del error se borraba a any Error
// Ahora: E propaga el tipo exacto del closure

extension Sequence {
    func mapValues<T, E>(
        _ transform: (Element) throws(E) -> T
    ) throws(E) -> [T] {
        var result: [T] = []
        for element in self {
            result.append(try transform(element))
        }
        return result
    }
}

// Si el closure lanza ParseError → mapValues lanza ParseError
// Si el closure no lanza (E == Never) → mapValues tampoco lanza
let raw = ["REC:Uno", "REC:Dos"]
let parsed = try raw.mapValues { try parseRecord(from: $0) }
// catch recibe ParseError directamente
