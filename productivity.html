Swift 6.0

Productividad · count(where:) · Pack iteration · Int128

Tres cosas que hacías con más código o con librerías externas. Ahora en la librería estándar.

count(where:) — sin array intermedio

filter { }.count creaba un array completo en memoria solo para contar. count(where:) hace un único recorrido sin alocar nada extra.

Swift 5.x — el problema

// Swift 5 — filter crea un array intermedio en memoria
let numbers = [1, 2, 3, 4, 5, 6]
let evenCount = numbers.filter { $0.isMultiple(of: 2) }.count
//                       ↑ crea [2, 4, 6, 8, 10] en memoria
//                                    luego cuenta — dos pasos, una alocación

Swift 6.x — la solución

// Swift 6 — un solo recorrido, sin alocación extra
let evenCount = numbers.count(where: { $0.isMultiple(of: 2) })
// Lee cada elemento una vez y cuenta los que cumplen la condición.
// Semánticamente claro: "cuenta los que cumplan esto"

Pack iteration — adiós a las sobrecargas manuales

En Swift 5, funciones como zip() o SwiftUI ViewBuilder tenían hasta 10 sobrecargas manuales — una por cada número de argumentos. Con parameter packs, una sola función cubre todos los casos.

Swift 5.x — el problema

// Swift 5 — una sobrecarga por cada número de argumentos (insostenible)
func allEmpty(_ a: [Any]) -> Bool { a.isEmpty }
func allEmpty(_ a: [Any], _ b: [Any]) -> Bool { a.isEmpty && b.isEmpty }
func allEmpty(_ a: [Any], _ b: [Any], _ c: [Any]) -> Bool {
    a.isEmpty && b.isEmpty && c.isEmpty
}
// ¿Y si alguien pasa 4, 5, 6 arrays? Hay que seguir añadiendo sobrecargas.
// Así está hecho zip(), SwiftUI ViewBuilder, etc — con hasta 10 sobrecargas.

Swift 6.x — la solución

// Swift 6 — una sola función para cualquier número de tipos distintos
func allEmpty<each T>(_ arrays: repeat [each T]) -> Bool {
    //         ──────                ──────────
    //     pack de tipos         uno por cada T
    for array in repeat each arrays {
        guard array.isEmpty else { return false }
    }
    return true
}

let a: [Int]    = []
let b: [String] = []
let c: [Double] = [3.14]

allEmpty(a, b)      // → true
allEmpty(a, b, c)   // → false (c no está vacío)
// Una sola función cubre 2, 3, 4... N argumentos de tipos completamente distintos

La regla: each T no es un tipo — es una colección de tipos. repeat expande el pack igual que un bucle for, pero en tiempo de compilación.

Int128 / UInt128 — sin librerías externas

Antes necesitabas una librería de terceros o hacer malabares con dos Int64. Ahora Int128 y UInt128 son tipos de primera clase en la librería estándar.

Swift 5.x — el problema

// Swift 5 — sin Int128 nativo
// Opciones: librería BigInt, Foundation.Decimal (coma flotante, no entero),
// o implementarlo tú mismo con dos Int64. Ninguna era elegante.
import Foundation
let bigNumber = Decimal(string: "170141183460469231731687303715884105727")!
// Decimal es coma flotante, no entero. No es lo mismo.

Swift 6.x — la solución

// Swift 6 — tipos nativos de primera clase
let bigA: Int128  = 170_141_183_460_469_231_731_687_303_715_884_105_727
let bigB: UInt128 = 340_282_366_920_938_463_463_374_607_431_768_211_455

// Soportan todos los operadores normales: +, -, *, /, Comparable, Hashable...
let suma = bigA + 1
let esMayor = bigA > Int128.zero

print(bigA)  // 170141183460469231731687303715884105727
