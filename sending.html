Swift 6.0

sending · Cruzar fronteras de actores

La palabra clave que permite pasar un valor a otro actor garantizando que nadie más lo retiene. Sin necesidad de conformar Sendable.

Antes — el compilador no podía verificar la transferencia

En Swift 5, pasabas objetos entre actores y el compilador no sabía si alguien más los retenía. Para cruzar fronteras de actores necesitabas conformar Sendable, lo que requería o bien un trabajo manual considerable o bien hacer trampa con @unchecked Sendable.

Swift 5.x — el problema

// Swift 5.x — el compilador no podía verificar esto
actor DataProcessor {
    func process(_ data: LargeData) async {
        // ¿Está data siendo usada desde otro contexto? No se sabe.
        await heavyWork(data)
    }
}

// Había que conformar Sendable (o @unchecked Sendable haciendo trampa)
class LargeData: @unchecked Sendable {
    var buffer: [UInt8]
    init(_ buffer: [UInt8]) { self.buffer = buffer }
}

Ahora — sending garantiza transferencia exclusiva

Con sending en el parámetro, el caller le promete al compilador que transfiere la propiedad exclusiva del valor. El compilador verifica que nadie más retiene ese valor tras la llamada. No hace falta conformar Sendable.

Swift 6.0 — la solución

// Swift 6.0 — `sending` garantiza transferencia exclusiva
actor DataProcessor {
    // `sending` = el caller promete que nadie más retiene `data`
    // tras la llamada. El compilador lo verifica.
    func process(_ data: sending LargeData) async {
        await heavyWork(data)
    }
}

// Ya no necesita ser Sendable
class LargeData {
    var buffer: [UInt8]
    init(_ buffer: [UInt8]) { self.buffer = buffer }
}

// Uso correcto: tras pasarlo, no puedes usarlo más
let processor = DataProcessor()
let myData = LargeData([1, 2, 3])
await processor.process(myData)
// myData ya no accesible aquí — el compilador lo garantiza

La regla: sending es diferente de Sendable. Sendable dice "este tipo es seguro para compartir en concurrencia". sending dice "este valor concreto lo transfiero en exclusiva, pase lo que pase con el tipo".

Reglas de sending

sending puede aparecer en parámetros y en resultados. En cada caso el contrato es distinto: quien llama pierde el acceso, o quien devuelve transfiere la propiedad.

PosiciónContrato
sending en parámetroel caller pierde el acceso al valor tras la llamada
sending en resultadoel callee transfiere la propiedad exclusiva al caller
sin sendingel tipo necesita conformar Sendable para cruzar fronteras
